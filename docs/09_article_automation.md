# 記事投稿自動化

## 概要

Claude Code に microCMS MCP を接続することで、記事生成から microCMS への投稿までを自動化する。
SEO を最大化するため、本文には箇条書き・表・FAQ・画像挿入マーカーを使用する。

---

## MCP 設定

`.claude/settings.json` に以下を設定済み（`.gitignore` で除外済み）：

```json
{
  "mcpServers": {
    "microcms": {
      "command": "npx",
      "args": ["-y", "microcms-mcp-server@latest"],
      "env": {
        "MICROCMS_SERVICE_ID": "it-no-migiude",
        "MICROCMS_API_KEY": "<書き込み権限付きAPIキー>"
      }
    }
  }
}
```

> APIキーは GET / POST / PUT / PATCH 権限が必要。DELETE は不要。

## microCMS コンテンツスキーマ（articles）

| フィールドID | 型 | 備考 |
|---|---|---|
| title | テキスト | 必須 |
| slug | テキスト | 必須。英数字ハイフン区切り |
| body | リッチエディタ（HTML） | 必須 |
| thumbnail | 画像 | Gemini/Imagen で生成 |
| category | コンテンツ参照（categories） | 必須 |
| tags | テキスト | カンマ区切り |
| metaDescription | テキスト | 80〜120字 |

## 記事フォーマット（SEO最適化版）

### タイトルパターン
`メインキーワード——補足説明` の形式（全角ダッシュ「——」で区切る）

```
例:
SharePointの使い方——社内ポータルとして活用するための基本設定
Slackの使い方——通知疲れを防ぐチャンネル設計と運用ルール
```

### 本文構成

- 文字数: 2,500〜4,000字（箇条書き・表が増えた分、上限を引き上げ）
- 見出し階層: **H2**（大テーマ）+ **H3**（細分化）の2段構造
  - H1・H4以下は使わない（H1はページタイトルで自動付与される）
- セクション区切り: 各 H2 の前に `<hr>` を挿入
- 締め: 「まとめ」H2 → 「よくある質問」H2（FAQ）→ 無料相談への CTA

### 全体テンプレート

```html
<p>冒頭の課題提起 2〜3段落（見出しなし）</p>

<!-- 図解マーカー: 記事内容を1枚で表現する図解。必要に応じて挿入 -->
<!-- IMAGE: hero-summary | 記事全体のイメージを表すフラットイラスト -->

<hr>
<h2>大テーマA</h2>
  <h3>細分化トピックA-1</h3>
  <p>段落...</p>
  <ul>
    <li>箇条書き項目1</li>
    <li>箇条書き項目2</li>
  </ul>

<hr>
<h2>大テーマB（比較・違いがあるテーマ）</h2>
  <h3>各オプションを比較する</h3>
  <table>
    <thead>
      <tr><th>項目</th><th>選択肢A</th><th>選択肢B</th></tr>
    </thead>
    <tbody>
      <tr><td>料金</td><td>月額○円</td><td>月額○円</td></tr>
    </tbody>
  </table>

<hr>
<h2>まとめ：〇〇は〇〇が重要</h2>
<p>本文のキーポイントを3〜4行で要約</p>

<hr>
<h2>よくある質問</h2>
  <h3>Q. 質問文1？</h3>
  <p>A. 回答...</p>
  <h3>Q. 質問文2？</h3>
  <p>A. 回答...</p>

<hr>
<p>締めの段落。「30分の無料相談でご状況を聞かせてください」につなげる</p>
```

### 使用可能なHTMLタグ

| タグ | 用途 |
|---|---|
| `<p>` | 段落 |
| `<h2>` `<h3>` | 見出し |
| `<hr>` | セクション区切り |
| `<ul>` `<ol>` `<li>` | 箇条書き・番号リスト |
| `<table>` `<thead>` `<tbody>` `<tr>` `<th>` `<td>` | 比較表 |
| `<strong>` | 強調（多用しない） |
| `<a href>` | リンク（内部リンク推奨） |

### 図解マーカー（画像自動生成用）

本文中に図解を挿入したい位置にHTMLコメントを置く。後処理スクリプトが Imagen API で画像を生成・差し替える。

```html
<!-- IMAGE: {slug-suffix} | {画像のプロンプト} -->
```

- `slug-suffix`: 記事内で一意なID（例: `channel-naming-rules`）。最終的に `${article-slug}-${slug-suffix}.png` で保存
- プロンプト: 何を描いた画像か日本語で短く記述（自動的に英語に翻訳してImagen に渡す）

#### 図解の使い所
- 記事冒頭（ヒーロー直下）に1枚: 記事の全体像
- 比較・分類セクションに1枚: 概念図やフロー
- **1記事に最大3枚まで**。多すぎると読みにくい

### 箇条書き・表の使い方

- **箇条書きで書くべき箇所**: 手順、メリット・デメリット、チェックリスト、要件一覧
- **表で書くべき箇所**: 2つ以上の選択肢の比較、料金プラン、機能対応表
- 段落で書ける内容を無理にリスト化しない。リスト3項目以下なら段落のほうが読みやすいこともある

### FAQセクション

- 「まとめ」の後に必ず配置
- Q&A は3〜5問
- Q.は具体的に。読者が検索しそうな質問文をそのまま使う（例: 「Slackは何人から有料プランにすべき？」）
- 回答は2〜3文で簡潔に
- このセクションは詳細ページ側で構造化データ（Schema.org/FAQPage）として吐き出されるため、必ず `<h2>よくある質問</h2>` の見出し名で書く

### 内部リンク

- 本文中で関連する過去記事に最低1〜2本リンクする
- リンク先のslugは事前に microCMS から既存記事リストを取得して確認
- `<a href="/articles/{slug}">アンカーテキスト</a>` 形式
- アンカーテキストは記事タイトルそのものではなく、文脈に合わせて自然に書く

### tags

カンマ区切り、5個前後の日本語キーワード

```
例: "Slack, チャンネル設計, 通知管理, 業務効率化, 社内コミュニケーション"
```

### metaDescription

疑問文や問いかけで始まるパターンが多い。80〜120字。

```
例: "Slackの通知疲れに悩んでいませんか？30名規模の中小企業を想定し、チャンネル命名規則・@channelの使い分け・絵文字リアクション運用まで、定着する設計と運用ルールを具体的に解説します。"
```

### slug

英数字ハイフン区切り（kebab-case）、25字以内目安

```
例: slack-channel-design-notification-rules
```

## 自動化フロー

```
1. トピック・キーワード・カテゴリを Claude に伝える
2. Claude が本文を生成（H2+H3構造、箇条書き・表・FAQ・図解マーカー入り）
3. microCMS MCP で下書き投稿
4. （任意）scripts/generate-article-images.mjs で図解を自動生成・差し替え
5. microCMS ダッシュボードでサムネ・図解確認・公開
```

```
以下のトピックで記事を書いて microCMS に下書き投稿してください。

トピック: [テーマ]
カテゴリ: [AI活用 / DX入門 / 業務効率化]
ターゲット: 中小企業・スタートアップの経営者・担当者
文字数: 2,500〜3,500字
```

## 守るべきこと

- **被り厳禁**: microCMS の既存記事タイトルと意味的に重複していないか必ずチェック。slug も既存と被らないこと
- **CTA は自然に**: 「30分の無料相談」誘導は最後の段落のみ
- **凡庸な表現を避ける**: 「〜と言えるでしょう」「〜が大切です」みたいなフワッとした締めは禁止。具体的な数字・固有名詞・実例を盛り込む
- **既存記事の文体に合わせる**: 「です・ます調」、淡々とした業務メディアトーン、煽らない
- **見出しに検索キーワードを入れる**: H2・H3 にも検索意図に近い語彙を盛り込む。リード文だけで終わらせない
